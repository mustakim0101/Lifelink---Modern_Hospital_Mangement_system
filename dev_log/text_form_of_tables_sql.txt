Below is an **updated, MSSQL-friendly, future-proof schema** that matches your roles + flows (Admin / IT Dept Admin / Doctors / Nurses / Patients / Applicants / Blood Donors) and adds the missing pieces: **RBAC, department-wise access, bed lifecycle, blood request → donor matching → donation → inventory**, plus auditing.

## 1) Key updates vs your old tables

### Replace “role” inside Users with proper RBAC

Old: `Users(role)`
New: `Roles`, `UserRoles`, optional `Permissions`, `RolePermissions`

This makes it easy later to add roles like **LabTech, Pharmacist, Receptionist** without changing Users.

### Split “Ward_Unit / Bed” into real bed management with departments

Old: Ward_Unit (unit_type, floor), Bed (status)
New: `CareUnits` + `Beds` + `BedAssignments`
So you can track **history** of who used which bed, and automatically free it when discharge happens.

### Admissions must include department + required care type

Add: `Admissions.department_id`, `care_level_requested`, `care_level_assigned`

### Blood bank becomes a module with requests + donor availability + donation + inventory movements

Old: Blood_Bank, Blood_Inventory, Blood_Donor, Blood_Request
New: add `DonorAvailability`, `BloodDonations`, `BloodInventoryTransactions`, and link requests to patients + dept + matching donors.
---

## 2) Updated tables, attributes (MSSQL ready)

### A) Identity / Roles / Accounts

#### 1. Users

* **user_id (PK, INT IDENTITY)**
* email (UNIQUE)
* password_hash
* full_name
* phone (nullable)
* date_of_birth (nullable)
* gender (nullable)
* account_status (Active/Frozen/Deleted)
* created_at, updated_at
* frozen_at (nullable)
* frozen_by_user_id (FK → Users.user_id, nullable)

#### 2. Roles

* **role_id (PK)**
* role_name (UNIQUE)
  Examples: Admin, ITWorker, Doctor, Nurse, Patient, Donor, Applicant

#### 3. UserRoles

* **user_id (PK, FK → Users)**
* **role_id (PK, FK → Roles)**
* assigned_at
* assigned_by_user_id (FK → Users, nullable)

> (Many-to-many: user can be Patient + Donor, or Nurse + Donor)

#### 4. Permissions (optional but future proof)

* **permission_id (PK)**
* permission_code (UNIQUE) (e.g., FREEZE_USER, APPROVE_APPLICATION)

#### 5. RolePermissions (optional)

* **role_id (PK, FK → Roles)**
* **permission_id (PK, FK → Permissions)**

#### 6. AuditLogs (highly recommended)

* **audit_id (PK)**
* actor_user_id (FK → Users)
* action_type (e.g., FREEZE_ACCOUNT, ASSIGN_BED, APPROVE_APP)
* entity_name (e.g., Admissions)
* entity_id (INT or UNIQUEIDENTIFIER)
* details (NVARCHAR(MAX))
* created_at
---

### B) Organization (Departments + IT scope)

#### 7. Departments

* **department_id (PK)**
* dept_name (UNIQUE)  (Cardiology, Neurology, etc.)
* is_active

#### 8. DepartmentAdmins (IT Workers scope)

* **user_id (PK, FK → Users)**  (must have ITWorker role)
* **department_id (PK, FK → Departments)**
* assigned_at

> Supports “IT worker can only search admitted patients in their dept”.
---

### C) Job applications flow (Applicant → Approved → Role assigned)

#### 9. JobApplications

* **application_id (PK)**
* user_id (FK → Users)  (the applicant’s user)
* applied_role_id (FK → Roles)
* applied_department_id (FK → Departments, nullable) (for doctor/nurse/it worker)
* status (Pending/Approved/Rejected)
* applied_at
* reviewed_by_user_id (FK → Users, nullable)
* reviewed_at (nullable)
* review_notes (nullable)

> When Approved → you insert into `UserRoles` and (if needed) create Doctor/Nurse/IT profile row.

---

### D) Staff profiles (medical staff)

#### 10. Doctors

* **doctor_id (PK, FK → Users.user_id)**
* department_id (FK → Departments)
* specialization
* license_number (UNIQUE)
* is_active

#### 11. Nurses

* **nurse_id (PK, FK → Users.user_id)**
* department_id (FK → Departments)
* ward_assignment_note (nullable)
* is_active
---

### E) Patients / clinical workflow

#### 12. Patients

* **patient_id (PK, FK → Users.user_id)**
* blood_group (A+, A-, B+, …, O-)
* emergency_contact_name
* emergency_contact_phone
* is_active

#### 13. Appointments

* **appointment_id (PK)**
* patient_id (FK → Patients)
* department_id (FK → Departments)
* doctor_id (FK → Doctors, nullable) (nullable until assigned)
* appointment_datetime
* status (Booked/Cancelled/Completed/NoShow)
* created_at
* cancelled_by_user_id (FK → Users, nullable)
* cancel_reason (nullable)

> Matches “patient books appointment”, “doctor can cancel/reject”.

#### 14. Admissions

* **admission_id (PK)**
* patient_id (FK → Patients)
* department_id (FK → Departments)
* admitted_by_doctor_id (FK → Doctors, nullable)
* admit_date
* discharge_date (nullable)
* status (Admitted/Discharged/Transferred/Cancelled)
* diagnosis_initial (nullable)
* care_level_requested (General/ICU/NICU/CCU)
* care_level_assigned (nullable)
* notes (nullable)

#### 15. MedicalRecords

* **record_id (PK)**
* patient_id (FK → Patients)
* admission_id (FK → Admissions, nullable) (so outpatient records also possible)
* created_by_user_id (FK → Users)
* record_datetime
* diagnosis
* treatment_plan
* notes (nullable)

#### 16. Prescriptions

* **prescription_id (PK)**
* patient_id (FK → Patients)
* admission_id (FK → Admissions, nullable)
* doctor_id (FK → Doctors)
* created_at
* status (Active/Stopped/Completed)

#### 17. PrescriptionItems

* **item_id (PK)**
* prescription_id (FK → Prescriptions)
* medication_name
* dosage
* frequency
* duration_days
* instructions (nullable)

> More future-proof than a single “medication” field.
--

### F) Bed / Ward / ICU management (department-wise)

#### 18. CareUnits

* **care_unit_id (PK)**
* department_id (FK → Departments)
* unit_type (Ward/ICU/NICU/CCU)
* floor (nullable)
* unit_name (nullable)
* is_active

#### 19. Beds

* **bed_id (PK)**
* care_unit_id (FK → CareUnits)
* bed_code (e.g., ICU-01) (UNIQUE within unit)
* status (Available/Occupied/Maintenance/Reserved)
* is_active

#### 20. BedAssignments  ✅ (this is what makes it “automatic free + history”)

* **assignment_id (PK)**
* admission_id (FK → Admissions)
* bed_id (FK → Beds)
* assigned_by_user_id (FK → Users)  (IT worker)
* assigned_at
* released_at (nullable)
* released_by_user_id (FK → Users, nullable)
* release_reason (Discharge/Transfer/Cancel/Maintenance)

Rules:

* Only **one active assignment** per admission (released_at IS NULL)
* Only **one occupied bed** at a time (bed can’t have 2 active assignments)

Nurses dashboard: join Admissions → BedAssignments(active) → Beds.
---

### G) Blood bank module (requests, donors, inventory, matching)

#### 21. BloodBanks

* **blood_bank_id (PK)**
* bank_name
* location
* is_active

#### 22. BloodInventory

* **inventory_id (PK)**
* blood_bank_id (FK → BloodBanks)
* blood_group (A+, A-, …)
* component_type (WholeBlood/Plasma/Platelets/RBC)  ✅ future-proof
* units_available (INT)
* last_updated_at

Unique constraint: (blood_bank_id, blood_group, component_type)

#### 23. BloodRequests

* **request_id (PK)**
* patient_id (FK → Patients)
* admission_id (FK → Admissions, nullable)
* department_id (FK → Departments)
* requested_by_user_id (FK → Users)  (patient or doctor)
* blood_group_needed
* component_type
* units_required
* urgency (Normal/Urgent/Emergency)
* status (Pending/Matched/Approved/Fulfilled/Rejected/Cancelled)
* request_date
* notes (nullable)

#### 24. DonorProfiles

* **donor_id (PK, FK → Users.user_id)**
* blood_group
* last_donation_date (nullable)
* is_eligible (BIT)
* notes (nullable)

#### 25. DonorHealthChecks (for weight/temp requirement)

* **check_id (PK)**
* donor_id (FK → DonorProfiles)
* check_datetime
* weight_kg
* temperature_c
* hemoglobin (nullable)
* notes (nullable)
* checked_by_user_id (FK → Users, nullable)

#### 26. DonorAvailability

* **availability_id (PK)**
* donor_id (FK → DonorProfiles)
* week_start_date
* is_available (BIT)
* max_bags_possible (INT)
* updated_at

#### 27. BloodDonations

* **donation_id (PK)**
* donor_id (FK → DonorProfiles)
* blood_bank_id (FK → BloodBanks)
* donation_datetime
* blood_group
* component_type
* units_donated
* recorded_by_user_id (FK → Users)  (IT worker)
* linked_request_id (FK → BloodRequests, nullable)  (if donation was for a request)

#### 28. BloodInventoryTransactions  ✅ (most future-proof part)

* **txn_id (PK)**
* blood_bank_id (FK → BloodBanks)
* donation_id (FK → BloodDonations, nullable)
* request_id (FK → BloodRequests, nullable)
* blood_group
* component_type
* units_change (+/-)
* reason (Donation/Fulfillment/Adjustment/Expired)
* created_at
* created_by_user_id (FK → Users)

> Inventory becomes reliable: when donation happens → insert +3; when request fulfilled → insert -units.

#### 29. BloodRequestMatches (donor selection list)

* **match_id (PK)**
* request_id (FK → BloodRequests)
* donor_id (FK → DonorProfiles)
* match_score (nullable)
* status (Suggested/Notified/Accepted/Declined/Completed)
* created_at
* responded_at (nullable)

Patients can “see who will be donor after donor selects” → show only Accepted.
---

### H) Notifications (IT worker sends universal notice by blood type)

#### 30. Notifications

* **notification_id (PK)**
* title
* message
* target_type (Role/Department/BloodType/SpecificUser)
* created_by_user_id (FK → Users)
* created_at

#### 31. NotificationTargets

* **notification_id (PK, FK → Notifications)**
* **user_id (PK, FK → Users)**

(mustgenerate target list at send-time based on blood type + availability.)
---

## 3) Relationships summary (connections)

* Users ↔ Roles: **many-to-many** via UserRoles
* DepartmentAdmins ties IT workers to departments
* Doctors/Nurses belong to exactly one department
* Patients are Users with Patient profile
* Appointment belongs to department + patient + optional doctor
* Admission belongs to patient + department + optional doctor
* BedAssignment connects Admission ↔ Bed (history)
* BloodRequest belongs to patient (+ optional admission) + department
* DonorProfiles is a User; availability & healthchecks per donor
* Donations add inventory via InventoryTransactions
* Fulfillment subtracts inventory via InventoryTransactions
* RequestMatches records donor acceptance flow
* Admin freeze/unfreeze: Users.account_status + logs
